## 問題の理解

### 🤔 なぜAIエージェントは「忘れる」のか？

- **コンテキストウィンドウの限界**: 全てのAIモデルには特定のコンテキストウィンドウがある
- **情報の蓄積**: 会話が長くなるにつれて、コンテキストウィンドウが満杯になる
- **ハルシネーションの発生**: コンテキストが満杯になると、モデルは幻覚を起こし始める
- **重要な情報の喪失**: 過去の重要な指示や決定事項を見失う

### 🎯 解決策の概要

1. **コンパクション**: 会話履歴を要約し、重要な要素を保持
2. **メモリツール**: ファイルベースの永続記憶システム
3. **構造化ノート**: プロジェクト状態の追跡と管理
4. **サブエージェント**: 専門化された複数エージェントの協調

## 1. コンパクション（圧縮）

### 🔄 自動コンパクション vs 手動コンパクション

#### 自動コンパクション
- **トリガー**: コンテキストウィンドウが90%以上使用された時
- **動作**: Claudeが自動的に会話履歴を要約
- **制限**: 何を保持するかはClaudeが決定

#### 手動コンパクション（推奨）
- **コマンド**: `@compact` または `compact` コマンド
- **利点**: 
  - 予期しないタイミングでの実行を防止
  - 保持したい情報を制御可能
  - カスタマイズ指示の追加可能

### 📝 実装手順

```bash
# 1. コンテキスト使用状況の確認
@context

# 2. 手動コンパクションの実行
@compact

# 3. カスタマイズ指示付きコンパクション
@compact 重要な設計決定とコードの変更履歴を保持してください
```

### ⚡ Claude Sonnet 4.5の新機能

- **200Kコンテキストウィンドウ**: より大きな作業領域
- **自動コンテキスト編集**: 古いツール呼び出し結果の自動削除
- **改善されたインテリジェンス**: 重要度の判断精度向上

## 2. メモリツール活用

### 💾 メモリツールの仕組み

- **ファイルベースシステム**: 作業環境内でのファイル作成・読み取り・削除・更新
- **永続記憶**: プロジェクト状態の保持
- **参照システム**: ファイル名とパスによるデータ取得

### 🎯 実装手順

#### 1. メモリへの保存
```bash
# 重要な指示をメモリに保存
@remember 新しいタスクを開始する前に、残りコンテキストを確認して警告を出すこと
```

#### 2. プロジェクトメモリファイルの作成
```markdown
# claude.md ファイル（プロジェクトルートに配置）

## プロジェクト概要
このプロジェクトは価格.comから車両データをスクレイピングするツールです。

## 技術仕様
- Python 3.12+
- BeautifulSoup4
- pandas
- requests

## 重要な指示
- エラーハンドリングを必ず実装
- レート制限を遵守
- ログ出力を詳細に記録
- データ検証を必須とする

## アーキテクチャ決定
- クラスベース設計を採用
- 設定ファイルによる柔軟な設定
- 出力ディレクトリの自動管理
```

### 📊 効果の比較

| 項目 | claude.mdなし | claude.mdあり |
|------|---------------|---------------|
| 機能完成度 | 基本的な機能のみ | 高度な機能まで実装 |
| UI品質 | シンプル | 現代的で洗練された |
| エラーハンドリング | 最小限 | 包括的 |
| ドキュメント | 不足 | 充実 |

## 3. 構造化ノートテイキング

### 📋 推奨ファイル構造

```
project_root/
├── claude.md           # プロジェクト指示
├── progress.md         # 進捗追跡
├── decisions.md        # アーキテクチャ決定
├── bugs.md            # バグ記録と解決策
└── requirements.md    # 要件仕様
```

### 📝 各ファイルの役割

#### progress.md
```markdown
# プロジェクト進捗

## 完了済みタスク
- [x] 基本スクレイピング機能
- [x] エラーハンドリング
- [x] ログ機能

## 進行中タスク
- [ ] スケジューラー機能
- [ ] データ分析機能

## 次のステップ
1. パフォーマンス最適化
2. ユーザーインターフェース改善
```

#### decisions.md
```markdown
# アーキテクチャ決定記録

## 2024-01-15: データベース選択
**決定**: SQLiteを使用
**理由**: 軽量で設定不要、単一ファイル管理
**影響**: 大規模データには不向き

## 2024-01-16: 非同期処理
**決定**: asyncioを採用
**理由**: 並列処理による高速化
**影響**: コード複雑度の増加
```

#### bugs.md
```markdown
# バグ記録と解決策

## 2024-01-15: メモリリーク
**症状**: 長時間実行でメモリ使用量が増加
**原因**: セッションオブジェクトの不適切な管理
**解決策**: コンテキストマネージャーの使用
**ステータス**: 解決済み
```

### 🔄 自動更新の設定

```bash
# プロジェクトメモリに追加
@remember 各タスク実行後に以下のファイルを更新すること：
- progress.md: 完了タスクと次のステップ
- decisions.md: 重要な技術決定
- bugs.md: 発見されたバグと解決策
```

## 4. サブエージェントアーキテクチャ

### 🏗️ アーキテクチャ概要

```
ユーザーリクエスト
       ↓
   リーダーエージェント（調整役）
       ↓
┌─────────────────────────┐
│  サブエージェント群      │
├─────────────────────────┤
│ • Web開発エージェント    │
│ • データ分析エージェント │
│ • テストエージェント     │
│ • ドキュメントエージェント│
└─────────────────────────┘
       ↓
   結果の統合・返却
```

### 🎯 各エージェントの役割

#### リーダーエージェント
- **役割**: タスクの分析と委譲
- **責任**: サブエージェント間の調整
- **ツール**: 全サブエージェントへのアクセス

#### 専門エージェント
- **Web開発エージェント**: フロントエンド・バックエンド開発
- **データ分析エージェント**: データ処理・分析・可視化
- **テストエージェント**: テスト作成・実行・品質保証
- **ドキュメントエージェント**: ドキュメント作成・更新

### ⚖️ メリットとデメリット

#### メリット
- **専門化**: 各エージェントが特定領域に特化
- **並列処理**: 複数タスクの同時実行
- **コンテキスト管理**: 各エージェントの独立したコンテキスト
- **スケーラビリティ**: 必要に応じてエージェント追加

#### デメリット
- **複雑性**: エージェント間の調整が必要
- **処理時間**: メッセージパッシングによる遅延
- **リソース使用**: 複数エージェントのメモリ使用
- **デバッグ困難**: 問題の原因特定が複雑

### 🛠️ 実装例

```python
# サブエージェントフレームワーク例
class TaskOrchestrator:
    def __init__(self):
        self.agents = {
            'web_dev': WebDevAgent(),
            'data_analyst': DataAnalystAgent(),
            'tester': TestAgent(),
            'documenter': DocumentAgent()
        }
    
    def process_request(self, request):
        # リクエスト分析
        task_type = self.analyze_request(request)
        
        # 適切なエージェントに委譲
        agent = self.select_agent(task_type)
        result = agent.execute(request)
        
        # 結果の統合
        return self.integrate_results(result)
```

## ベストプラクティス

### ✅ 推奨事項

1. **定期的なコンパクション**
   - コンテキスト使用率80%で手動実行
   - 重要な指示を明示的に保持

2. **メモリの活用**
   - claude.mdの継続的な更新
   - 重要な決定事項の記録

3. **構造化ノートの維持**
   - 定期的なファイル更新
   - 一貫したフォーマット

4. **段階的な導入**
   - 一度に全てを導入せず、段階的に実装
   - 効果を測定しながら調整

### ❌ 避けるべきこと

1. **過度なコンパクション**
   - 頻繁すぎる実行は逆効果
   - 重要な情報の喪失リスク

2. **メモリの過剰使用**
   - 不要な情報の蓄積
   - ファイルサイズの肥大化

3. **サブエージェントの乱用**
   - 単純なタスクでの使用は非効率
   - 適切なタスク分割の重要性

### 🎯 タスク別推奨手法

| タスクタイプ | 推奨手法 | 理由 |
|-------------|----------|------|
| 長時間開発 | コンパクション + 構造化ノート | コンテキスト管理と進捗追跡 |
| 反復的開発 | メモリツール + 構造化ノート | 状態保持と決定記録 |
| 複雑なプロジェクト | サブエージェント + 全手法 | 専門化と効率化 |
| 単純なタスク | 基本メモリツール | シンプルさを重視 |
